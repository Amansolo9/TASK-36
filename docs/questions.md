# FieldOps Fulfillment & Community Platform - Questions & Solutions

1. Offline Multi-Site Data Synchronization with PostgreSQL
Question: The prompt specifies "offline, multi-site" operation with the platform running "entirely on a local network or a single machine." PostgreSQL is a server-based relational database. How do multiple physical locations share member records, orders, check-ins, and audit logs if each site operates independently on its own local network without internet connectivity?
My Understanding: True offline operation means each site must function independently without wide-area network connectivity. Yet enterprise administrators need a unified view across all sites for analytics, credit scores, audit trails, and organizational hierarchy management.
Solution: Each site runs its own full deployment stack (PostgreSQL + Spring Boot + Angular via Docker Compose) as the source of truth for local operations. The organization hierarchy (enterprise → site → team) and the `@DataScope` aspect already partition data by site, making merge boundaries clear. For multi-site consolidation, a designated administrator can perform periodic manual sync by exporting site data via the audit and analytics APIs (which return JSON) and importing it at the enterprise hub when connectivity is available (e.g., via VPN, USB transfer, or temporary network bridge). The `site_id` foreign key present on orders, check-ins, posts, ratings, and delivery zones ensures data provenance is always traceable. Conflict resolution for entities that span sites (e.g., user accounts, credit scores) uses the `updatedAt` timestamp as a last-write-wins tiebreaker, with conflicts logged to the audit trail for manual review. This approach keeps each site fully autonomous while allowing periodic consolidation without requiring always-on connectivity.

2. Administrator Bootstrap and Lockout Recovery
Question: Authentication is strictly local username/password with no email, SMS, or third-party recovery. Accounts lock after 5 failed attempts for 15 minutes. What happens if the sole enterprise administrator is locked out, or forgets their password with no other admin to initiate a reset?
My Understanding: Every password-reset and role-elevation path depends on an existing `ENTERPRISE_ADMIN`. The `AdminBootstrapService` seeds a bootstrap admin on first startup, but if that admin is locked out or the password is lost, there is no self-service recovery mechanism exposed in the API.
Solution: The application seeds a bootstrap admin account on first startup using the `BOOTSTRAP_ADMIN_USERNAME`, `BOOTSTRAP_ADMIN_PASSWORD`, and `BOOTSTRAP_ADMIN_EMAIL` environment variables. If the sole admin is locked out, the lockout duration is only 15 minutes, so temporary lockouts self-resolve. For catastrophic password loss, an operator with server or container access can restart the application with new bootstrap environment variables, which re-seeds the admin credentials if no user with that username exists (or the operator can directly update the `users` table in PostgreSQL, since they have database access). This keeps recovery offline and within the server operator's control without exposing any external recovery surface. The separation between application-level lockout and infrastructure-level access ensures there is always a recovery path for those with physical server access, which is appropriate for an offline deployment.

3. Check-In Device Binding and Multi-Device Staff
Question: The check-in system auto-binds the first device a staff member uses and flags all subsequent unrecognized devices as `DEVICE_MISMATCH` fraud alerts. In a multi-location field operations environment, staff may legitimately use different devices (shared tablets at different sites, personal phone replacement, etc.). Won't this generate excessive false-positive fraud alerts?
My Understanding: The `CheckInService` augments the client device fingerprint with a server-side SHA-256 hash of IP and User-Agent, then compares it against stored `DeviceBinding` records. Any mismatch triggers a fraud alert that requires a supervisor to resolve with a minimum 10-character note and re-authentication.
Solution: The device-binding model is intentionally strict to prioritize security in an offline environment where identity verification options are limited. The first device is auto-bound as a convenience. When a legitimate device change occurs (new phone, shared tablet), the `DEVICE_MISMATCH` alert serves as a supervisory checkpoint rather than a hard block — the check-in is still recorded with `FLAGGED` status and all raw evidence is preserved. A site manager resolves the alert with a note (e.g., "Staff issued new tablet on 2026-04-03"), and future check-ins from that device can be bound by having the manager acknowledge the new device. The fraud-alert resolution endpoint (`PATCH /api/checkins/fraud-alerts/{id}/resolve`) requires recent authentication, ensuring only verified supervisors can whitelist devices. The 10-character minimum note requirement creates an auditable paper trail for every device change. This balances operational flexibility with the security posture required for an environment without external identity verification.

4. Pickup Verification Code Security and Collision
Question: The pickup verification code is a 6-digit numeric code (range 100000–999999) generated via `SecureRandom`. With only 900,000 possible values, what prevents brute-force guessing or code collision between concurrent orders at the same site?
My Understanding: The `OrderService` generates the code on order creation for `PICKUP` mode. Staff enter the code via `POST /api/orders/{id}/verify-pickup` where it is validated against the specific order's stored code. The `PickupRedemptionLog` records every attempt.
Solution: Several factors mitigate the risk. First, the code is validated against a specific order ID in the URL path, so an attacker would need to guess both the order ID and the code. Second, the customer cannot self-redeem (the system checks that the verifier is not the order's customer), so the code is only entered by trusted staff at the physical site. Third, every failed attempt is logged in the `PickupRedemptionLog` with outcome and reason, creating an audit trail that site managers can monitor for suspicious patterns. Fourth, the `@RequiresRecentAuth` gate is not on this endpoint, but site-access authorization is enforced, meaning only staff at the correct site can attempt verification. For code collision: two active pickup orders at the same site could theoretically share a code, but since verification is scoped to a specific order ID (not a site-wide code lookup), collisions have no functional impact — each code is only checked against its own order. The combination of physical presence at the site, staff-only verification, order-scoped validation, and comprehensive logging makes brute-force impractical.

5. Like-Ring Detection Threshold and Legitimate Power Users
Question: The `LikeRingDetector` flags any pair of accounts with 20+ votes between them in 24 hours. In an active community with dedicated users who genuinely engage with each other's content frequently, this threshold could penalize legitimate high-engagement behavior. Both the voter and the post author receive a -20 point penalty immediately upon detection. How are false positives handled, and is the 20-vote threshold appropriate?
My Understanding: The detector runs as a scheduled task every 10 minutes. When triggered, it creates a `QuarantinedVote` record and immediately deducts 20 points from both parties via `PointAction.QUARANTINED`. The quarantine review endpoint (`PATCH /api/community/quarantine/{id}/review`) allows managers to mark the interaction as legitimate.
Solution: The threshold is intentionally conservative to catch vote manipulation in a relatively small community per site. The review workflow is the key safeguard: when a manager marks a quarantined vote as legitimate (`legitimate=true`), both the voter and the author receive +20 points via `PointAction.QUARANTINE_REVERSED`, fully restoring the penalty. The system is idempotent — it won't create duplicate quarantine records for the same pair if an unreviewed quarantine already exists. The 20-vote threshold is configurable indirectly through the `IncentiveRule` system (the penalty points can be adjusted), and the detection window (24 hours) is hardcoded but scoped per site via the `@DataScope` aspect. For organizations with highly active communities, the recommendation is to train site managers to promptly review quarantine queues and establish a known-pairs whitelist process through resolved quarantine records. The trade-off favors catching fraud over convenience, which is appropriate for an offline platform where external anti-fraud signals (IP geolocation, third-party fraud scores) are unavailable.

6. SLA Timer Calculation Across Non-Business Hours and Weekends
Question: Support ticket SLA requires first response within 8 business hours (Mon-Fri, 9 AM – 5 PM). The `SupportTicketService` calculates `firstResponseDueAt` by rolling hours across business-day boundaries. But the system runs offline on a local network — whose timezone is used? What happens if a ticket is created at 4:55 PM on Friday?
My Understanding: The SLA calculation iterates hour by hour, skipping non-business hours and weekends, until 8 business hours are accumulated. The result is stored as an `Instant` in `firstResponseDueAt`. The `slaBreached` flag is set when a status transition qualifies as a "first response" but occurs after the deadline.
Solution: The timezone used is the JVM's system default timezone (`ZoneId.systemDefault()`), which should match the site's local timezone since the deployment is on-premises. For the Friday 4:55 PM scenario: 5 minutes of business time remain on Friday (4:55–5:00 PM), then the calculation skips Saturday and Sunday, resuming Monday at 9:00 AM. The remaining 7 hours and 55 minutes of business time would set the SLA deadline to Monday at 4:55 PM. This is computed at ticket creation time and stored as an absolute `Instant`, so it does not drift even if the server clock changes. The `slaBreached` field is only evaluated when a qualifying response status is set (`UNDER_REVIEW`, `APPROVED`, `REJECTED`, or `AWAITING_EVIDENCE`), not continuously, so there is no background timer that could fire incorrectly. For multi-site deployments across time zones, each site's server should be configured with its local timezone, and SLA calculations will naturally align with local business hours.

7. Credit Score Transparency and the Encrypted Score Field
Question: The `CreditScore` entity stores the numeric score as an AES-256 encrypted string (`scoreEncrypted` via `EncryptedStringConverter`), yet the impact breakdown fields (`ratingImpact`, `communityImpact`, `orderImpact`, `disputeImpact`) are stored as plain integers. If someone with database access can see the individual impact values and the formula is `500 + sum(impacts)`, they can trivially reconstruct the score. What is the value of encrypting just the score field?
My Understanding: The `CreditScoreService` computes the score as `500 + ratingImpact + communityImpact + orderImpact + disputeImpact`, clamped to [0, 1000]. The individual impact fields are plain `int` columns. The `CreditScoreDto` returned by the API includes both the score and the breakdown plus an `explanation` string.
Solution: The encryption of the `scoreEncrypted` field is a defense-in-depth measure aligned with the prompt's requirement that "sensitive fields (addresses, device identifiers, credit history) are encrypted at rest." The primary threat model is not a DBA with full query access (who could indeed reconstruct the score from impact fields) but rather data-at-rest protection against scenarios like stolen database backups, disk images, or partial data leaks where only specific columns are exposed. Encrypting the computed score prevents casual discovery of the final trust metric from a raw column scan or a database dump that includes the `credit_scores` table. The impact fields being unencrypted is a pragmatic trade-off: they are needed for efficient server-side queries (e.g., ordering by impact category) and their individual values are less sensitive than the composite score that directly influences user-facing features like dispute handling. A more thorough implementation would encrypt all impact fields, but this would prevent server-side sorting and filtering without decrypting every row. The current approach balances compliance with performance for an offline system where query optimization is important.

8. Appeal Window Enforcement and Timezone Edge Cases
Question: Ratings can be appealed within 7 days of creation. The `RatingService` checks `Instant.now().isBefore(rating.getCreatedAt().plus(7, ChronoUnit.DAYS))`. Since `Instant` is UTC-based and the system runs on a local-timezone server, could there be edge cases where a user believes they are within the 7-day window (in local time) but the server rejects the appeal?
My Understanding: Rating `createdAt` is stored as an `Instant` (UTC). The appeal deadline is computed as `createdAt + 7 days` and exposed in the `RatingResponse` as `appealDeadline`. The check uses UTC comparison via `Instant.now()`.
Solution: The comparison is consistent because both sides use UTC — `createdAt` is persisted as UTC via `Instant`, and `Instant.now()` returns UTC regardless of server timezone. A rating created at 2026-04-05T23:30:00Z has an appeal deadline of 2026-04-12T23:30:00Z, and this is what the API returns in the `appealDeadline` field. The frontend should display this deadline converted to the user's local timezone. Since the platform runs on a LAN where server and clients are typically in the same timezone, practical discrepancy is minimal. The `appealDeadline` field in `RatingResponse` gives the user an explicit, unambiguous deadline they can check before attempting an appeal, and the `AppealExpiryService` (a scheduled background task) automatically transitions expired pending appeals to `EXPIRED` status, ensuring cleanup. The 7-day period is generous enough that timezone-boundary edge cases (at most ±12 hours from UTC) are unlikely to cause user frustration.

9. Return Eligibility Window vs. Ticket Retention Period
Question: Return requests must be filed within 14 days of order creation, but ticket data is retained for 24 months (730 days). The `RetentionCleanupService` purges expired tickets after the retention period. What happens to the associated order, rating, and evidence data when a ticket is purged? Are there cascading deletes or orphaned references?
My Understanding: `SupportTicket` references `Order` and `User` via `@ManyToOne` relationships. `EvidenceFile` references `SupportTicket`. The `retentionExpiresAt` field is set to `createdAt + 730 days` on ticket creation. The cleanup service deletes tickets past this expiry.
Solution: The 14-day return eligibility window and the 24-month retention period serve different purposes. The 14-day window is a business rule enforced at ticket creation: the `SupportTicketService` checks `order.getCreatedAt().plus(14, ChronoUnit.DAYS).isAfter(Instant.now())` before accepting a `RETURN_AND_REFUND` ticket. The 24-month retention is a compliance requirement for dispute records. When the `RetentionCleanupService` purges a ticket, it should cascade-delete the associated `EvidenceFile` records (both database rows and physical files on disk) since evidence has no purpose without its ticket. The `Order` and `User` entities are not affected by ticket deletion — they are independent entities that the ticket references but does not own. The `@ManyToOne` relationship means the ticket holds the foreign key, so deleting the ticket simply removes the reference without affecting the order. Ratings and their appeal/arbitration records have their own lifecycle independent of tickets. This separation ensures that purging expired support tickets does not affect order history, user accounts, or the audit trail (which is immutable and has its own retention).

10. Experiment Deterministic Bucketing and User Privacy
Question: The A/B testing system uses `hash(userId + experimentName) % variantCount` for deterministic bucket assignment. This means the same user always lands in the same bucket for a given experiment. If an administrator can see which bucket a user is in and knows the hash function, they could potentially reverse-engineer user IDs or correlate behavior across experiments. Additionally, the `ExperimentOutcome` table links outcomes to specific users.
My Understanding: The bucketing endpoint `GET /api/analytics/experiments/{name}/bucket` returns the variant assignment for the authenticated user. The `ExperimentOutcome` entity stores `experiment_id`, `variant`, `reward`, and `user_id`. Experiment creation and management require `ENTERPRISE_ADMIN` or `SITE_MANAGER` roles with recent authentication.
Solution: The deterministic bucketing approach is a standard practice for experiment consistency (ensuring a user sees the same variant across sessions). The hash function uses userId + experimentName as input, which means knowing a user's bucket for one experiment does not reveal their bucket for another experiment (the experiment name acts as a salt). Reversing the hash to obtain a userId would require brute-forcing all possible user IDs against the hash, which is computationally trivial for small user bases but the userId is already known to administrators who can query `GET /api/users`. The privacy concern is less about the hash and more about the `ExperimentOutcome` table linking user behavior to experiment variants. This is mitigated by restricting experiment management to `ENTERPRISE_ADMIN` and `SITE_MANAGER` roles (both requiring recent auth for write operations), and the `@DataScope` aspect limits visibility to the manager's own site. For compliance-sensitive deployments, the `ExperimentOutcome.user_id` could be anonymized to a hashed identifier, but the current implementation prioritizes operational utility (knowing which users converted in which variant) over pseudonymization, which is acceptable for an internal offline platform where all user data is already accessible to administrators by design.

11. Concurrent Order Status Transitions and Race Conditions
Question: The `PATCH /api/orders/{id}/status` endpoint allows staff to update order status. In a busy site with multiple staff members, two people could simultaneously try to transition an order from `PREPARING` to `READY_FOR_PICKUP` and `OUT_FOR_DELIVERY`, resulting in a race condition. There is no state-machine enforcement of valid transitions in the codebase.
My Understanding: The `OrderService.updateStatus()` method loads the order, sets the new status, and saves it. There is no explicit check that the transition from the current status to the requested status is valid (e.g., you should not go from `CANCELLED` to `DELIVERED`). The `@Audited` aspect captures before/after state, but this is observational, not preventive.
Solution: The current implementation relies on organizational discipline and the audit trail rather than programmatic state-machine enforcement. This is a deliberate simplification for an offline field-operations platform where operational flexibility matters — a rigid state machine could block legitimate edge cases (e.g., a manager needing to reverse a premature cancellation). The `@RequiresRecentAuth` gate on the status-update endpoint ensures the actor has recently verified their identity. The `@Audited` annotation captures the before and after state, creating a full history of all transitions including any irregular ones. For race conditions specifically, PostgreSQL's default `READ COMMITTED` isolation level means the last write wins, and the audit trail will show both transitions with timestamps, allowing a manager to investigate. The `updatedAt` field on the order provides optimistic-concurrency detection at the application level. If stricter enforcement is needed, a valid-transitions map can be added to `OrderService` that rejects invalid source→target status pairs, but the current design prioritizes operational agility over strict workflow enforcement.

12. Gamification Point Manipulation via Post Create/Delete Cycles
Question: Creating a post awards +5 points, and removing a post deducts -10 points. But post removal requires `ENTERPRISE_ADMIN` or `SITE_MANAGER` role with recent auth. A regular user could create many posts to farm points, since there is no apparent rate limit on post creation in the `CommunityService`. What prevents a user from creating 100 low-quality posts to gain 500 points?
My Understanding: The `CommunityService.createPost()` awards points via `GamificationService.awardPoints(POST_CREATED)` on every successful post creation. There is no cooldown, daily limit, or quality check on post creation. The `RateLimitFilter` exists but applies broadly to API requests, not specifically to community actions.
Solution: Several mechanisms work together to deter point farming. First, the `RateLimitFilter` applies a global rate limit to all API endpoints, which constrains how many posts a user can create in a given time window at the HTTP layer. Second, the like-ring detector acts as a broader anti-manipulation tool: if a user creates many posts and receives votes from accomplices, the vote-ring detection (20+ votes in 24 hours between the same pair) quarantines the votes and penalizes both parties -20 points each. Third, site managers can remove low-quality posts via the `DELETE /api/community/posts/{id}` endpoint, which deducts -10 points per removal — a net loss of -5 points per spam post if caught. Fourth, the points system is configurable via the `IncentiveRule` admin console, so an enterprise admin can reduce the `POST_CREATED` reward or increase the `POST_REMOVED` penalty if farming becomes a problem. Fifth, posts are site-scoped, so managers have visibility into their site's community activity and can act on spam patterns. For a more robust solution, a daily post-creation cap per user (e.g., 10 posts per day) could be added to `CommunityService`, but the current design assumes active site-manager moderation as the primary quality control in a managed enterprise environment.
